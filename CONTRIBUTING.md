## 📝 **Style Guide**

### **Writing Tone & Voice**
- **Professional but Accessible** - Write for both experts and newcomers
- **Action-Oriented** - Focus on practical, implementable insights
- **Evidence-Based** - Support claims with research, data, or examples
- **Inclusive Language** - Use welcoming, non-discriminatory language
- **Clear & Concise** - Avoid jargon and unnecessary complexity

### **Markdown Formatting Standards**

#### **File Structure**
```markdown
# Main Title (H1)
## Section Title (H2)
### Subsection Title (H3)
#### Detail Title (H4)

**Bold text** for emphasis
*Italic text* for definitions or foreign terms
`code` for technical terms or file names
```

#### **Content Organization**
- **Start with an overview** of what the document covers
- **Use clear headings** to organize information logically
- **Include practical examples** and real-world applications
- **End with next steps** or related resources
- **Cross-reference** related topics within the encyclopedia

#### **Links & References**
- **Internal links** to related encyclopedia content
- **External links** to authoritative sources and research
- **Consistent link text** that describes the destination
- **Verify all links** work and are accessible

### **File Naming Conventions**
- **Use lowercase** with hyphens for spaces: `ai-transformation-guide.md`
- **Be descriptive** but concise: `workforce-skills-assessment.md`
- **Include category** in filename when helpful: `financial-services-ai-case-study.md`
- **Avoid special characters** except hyphens and underscores

## 🔄 **Pull Request Process**

### **1. Before You Submit**

#### **Content Contributions**
- [ ] **Content is original** or properly attributed
- [ ] **Information is accurate** and up-to-date
- [ ] **Style guide** is followed consistently
- [ ] **Cross-references** are added to related content
- [ ] **Spelling and grammar** are checked

#### **Technical Contributions**
- [ ] **Code follows** project coding standards
- [ ] **Tests pass** (if applicable)
- [ ] **Documentation is updated** to reflect changes
- [ ] **No breaking changes** without proper deprecation
- [ ] **Performance impact** is considered

### **2. Creating Your Pull Request**

#### **PR Title Format**
```
[CONTENT] Add AI transformation case study for healthcare
[TOOL] Enhance AI ROI calculator with new metrics
[BUGFIX] Fix navigation issue in skills assessment
[ENHANCEMENT] Improve search functionality
```

#### **PR Description Template**
```markdown
## 🎯 What does this PR do?
Brief description of the changes and their purpose.

## 🔍 What was changed?
- List of specific changes made
- Files added, modified, or removed
- New features or improvements

## 🧪 How was it tested?
Description of how you tested your changes.

## 📸 Screenshots (if applicable)
Add screenshots for UI changes or new features.

## 🔗 Related Issues
Closes #123, Related to #456

## ✅ Checklist
- [ ] Content follows style guide
- [ ] Links are verified and working
- [ ] No breaking changes introduced
- [ ] Documentation is updated
```

### **3. Review Process**
- **Automated checks** will run on your PR
- **Community review** by maintainers and contributors
- **Feedback and suggestions** for improvements
- **Approval and merge** once requirements are met

## 🚫 **What Not to Contribute**

### **Content Restrictions**
- **Plagiarized content** or unlicensed materials
- **Outdated information** or obsolete methodologies
- **Promotional content** or commercial advertisements
- **Personal opinions** without supporting evidence
- **Content that violates** our code of conduct

### **Technical Restrictions**
- **Breaking changes** without proper deprecation
- **Untested code** or incomplete implementations
- **Security vulnerabilities** or unsafe practices
- **Performance regressions** without justification
- **Dependencies** that conflict with existing ones
